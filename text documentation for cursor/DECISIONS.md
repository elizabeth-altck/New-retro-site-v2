# История решений

## 2026-06-27

### Решение

Главная страница является центральной комнатой всего мира GIF WORLD.

### Причина

Именно с неё пользователь начинает исследование сайта.

### Последствия

Все остальные страницы должны ощущаться как продолжение этого мира, а не отдельные сайты.

---

## 2026-06-27

### Решение

Навигация должна происходить не только через меню, но и через объекты страницы.

### Причина

Исследование мира интереснее обычной навигации.

### Последствия

Любая новая страница может использовать двери, окна, картинки, GIF, предметы или комнаты как способ перехода.

---

## 2026-06-27

### Решение

Разрешается использовать разные стили оформления на разных страницах.

### Причина

Каждая страница представляет собой отдельную часть цифрового мира.

### Последствия

Страница рабочего стола, гостиницы и музыкальной комнаты могут значительно отличаться визуально, сохраняя общую атмосферу проекта.

---

## 2026-06-24

### Решение

Ввести принцип **«миры внутри объектов»**: декоративные предметы на странице при взаимодействии открывают не попап, а маленький самостоятельный мир со своим оформлением и интерактивом.

### Причина

Это усиливает ощущение исследования, любопытства и «живого» цифрового места — в духе старого интернета и Neocities, где клик по объекту открывает неожиданное.

### Последствия

- При проектировании страниц предпочтительно использовать предметы-контейнеры вместо стандартных модальных окон без атмосферы.
- Каждый такий мир может иметь собственный визуальный стиль, но должен ощущаться частью GIF WORLD.
- Конкретные объекты по страницам — в `IDEAS.md` (раздел «Миры внутри объектов»).

## 2026-06-24

### Решение

Макеты Figma (1920px) использовать только как **референс пропорций**, не переносить размеры буквально в CSS.

### Причина

Реальный viewport и читаемость на вебе отличаются от статичного макета; адаптация сохраняет дух, но не ломает вёрстку.

### Последствия

При сверке с макетом подгонять относительные соотношения (зазоры, крупность акцентов), а не копировать px один в один.

## 2026-06-24

Шапка главной: полупрозрачная тёмная подложка на всю ширину; заголовок **old net vibe** с жёсткой вишнёвой тенью и кремовым бликом; справа — **два слоя** одного PNG-сердца (тёмный offset + светлый).

### Причина

Соответствие retro-макету и ощущению «old net»; двойное сердце даёт объём без отдельного ассета.

### Последствия

- Текст тикера остаётся project-specific («old net core…»), не как в макете.
- Зазор шапка → контент через `--header-content-gap`, без уменьшения padding layout.
- Детали размеров и цветов — в `STYLE_GUIDE.md`.

## 2026-06-24

Первая реализация «мира внутри объекта» на главной — мини-игра **SIGNAL CATCH** в пунктирной зоне под заголовком **my site**; достижения пишутся в live events справа.

Следующие object-worlds (коробка, конверт, монитор) проектировать по тому же паттерну: свой UI внутри зоны + опциональная обратная связь в систему.

## 2026-06-28

### Решение

Коробка-коллаж на главной: **два PNG** (`closed` / `opened`) в `images/memory-box/`, открытие по **наведению** (на touch — тап). Не отдельный попап — смена состояния картинки в контейнере слева в `.core`.

### Причина

Визуал из Figma как коллаж; пользователь хочет оценять объект на сайте и легко заменять PNG при итерации дизайна.

### Последствия

- Справа от коробки — `#coreObjectsSlot` для других объектов.
- Зернистость и эффекты — через CSS на контейнер, без правки PNG.
- Внутренний «мир» коробки (gif, текст) — следующий этап, не блокирует смену внешних PNG.

## 2026-06-28

### Решение

В `#coreObjectsSlot` — **forecast.exe (Win98)**: окно «today's Forecast» + наложенное «НОВАЯ КОЛЛЕКЦИЯ» с меню File / Edit / Search / Close и кнопкой Look up; попапы через `#win98Popup`, события в `#eventsList`.

### Причина

Макет Figma и принцип object-world: не голый попап, а «живое» окно в стиле Win98 внутри центральной комнаты.

### Последствия

- Look up и меню пока дают атмосферные попапы, не реальную навигацию в коллекцию.
- Close скрывает окно коллекции визуально (`is-closed`), не удаляя из DOM.
- Дальнейший «мир коллекции» можно встроить в Look up или отдельную страницу.

## 2026-06-28

### Решение

**Улитка** — отдельный object в `.core` (`#snailDragWrap`), не внутри слота: **ползает сама** по оливковому блоку; пользователь может **перетащить** — после отпускания продолжает ползать **от новой позиции**, без возврата назад.

### Причина

Живой, смешной персонаж сайта; перетаскивание — вмешательство пользователя, не основной способ движения. Возврат на старую позицию после drag созвучен с «отменой» и ломает ощущение присутствия.

### Последствия

- Ассет: `images/snail.png`, размер ~**320px** (как в макете).
- Hover: мягкое свечение + блёстки на раковине; без резкого блика.
- Тап: shy → иногда `snail.exe` → каждый 5-й тап «полёт»; ползание **не** по тапу.
- События (pet, kick, flight) — в `#eventsList`.
- Ползание и drag — **по всему viewport** (`position: fixed`, без лимита `.core`); стартовая позиция — между site TODO scrollport и forecast.exe, не поверх блока заметок.

## 2026-06-28

### Решение

Целевая **V1 главной** — не «идеальный сайт», а **полный вертикальный scroll по макету**: все основные секции свёрстаны (note, intro, image of day, the snail), навигация ведёт хотя бы на заглушки страниц.

### Причина

Много концептов и желание работать постоянно; без «готовой когда-нибудь версии» проект остаётся вечным кусочным прототипом. Глубина (tamagotchi, комнаты персонажей, коллекция) — **после** цельного scroll.

### Последствия

- Приоритет: закрыть scroll главной по макету, затем gallery;
- виджет улитки остаётся на главной; переход в «комнату Жанны» — отдельный этап;
- tamagotchi / блок «the snail» — секция на главной + будущие страницы Жанна / Анна;
- пасхалки и полировка object-worlds — фоновый режим после V1 scroll.
